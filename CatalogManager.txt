1. Database and Architecture Design Changes
1.1 Catalog Storage Architecture
We propose a self-hosting catalog system with page-based storage that integrates
seamlessly with the existing RookDB's existing infrastructure. The proposed Catalog
Storage Architecture is as follows:
database/
├── global/
│ ├── catalog_pages/ # NEW: Page-based catalog storage
│ │ ├── pg_database.dat # System catalog: databases
│ │ ├── pg_table.dat # System catalog: tables
│ │ ├── pg_column.dat # System catalog: columns
│ │ ├── pg_constraint.dat # System catalog: constraints
│ │ ├── pg_index.dat # System catalog: indexes
│ │ └── pg_type.dat # System catalog: data types
│ └── catalog.json # DEPRECATED: Legacy format (migration support)
└── base/
└── {database}/
└── {table}.dat # User table data files
To justify this structure:
● Page-based storage enables buffer manager integration for efficient caching,
● System catalogs provide self-hosting capabilities similar to PostgreSQL,
● Separation of concerns between system metadata, and
● Scalability supports large numbers of databases, tables, and constraints
1.2 Page Layout for Catalog Pages
For catalog pages, we’ll continue following the slotted page layout consistent with
RookDB's existing page structure, for the following reasons:
● Consistency: Matches existing RookDB page layout
● Efficient updates: Item IDs allow tuple relocation without changing references
● Space efficiency: Variable-length tuples with compact storage
● Buffer compatibility: Standard page size (8KB) works with buffer manager

1.4 Buffer Manager
We further propose to integrate the catalog system with the existing Buffer Manager
through a Catalog Page Cache; via the following key
integration points:
● Catalog Page Allocation:
○ System catalog tables use reserved page range
(OID < 10000)
○ Special page type identifier in page header
● Cache Invalidation:
○ Catalog cache entries invalidated on DDL ops
○ Write-through updates ensure consistency
1.5 Constraint System Architecture
The architecture supports the following constraints on attribute columns:
● Primary Key: Unique index, NOT NULL enforced, supports composite keys.
● Foreign Key: Referential integrity, cascading actions (CASCADE, SET NULL,
RESTRICT, NO ACTION), supports composite keys.
● Unique: Creates unique index, allows NULLs (unless
combined with NOT NULL).
● NOT NULL: Column-level, enforced on INSERT/UPDATE.
1.6 OID (Object Identifier) System
To uniquely identify all database objects, we implement an OID
system. Stored in a special system page:
database/global/pg_oid_counter.dat with atomic increment
operations and next available OID (UINT32) stored in page 0
header for efficiency.
1.7 Justification
To justify all the proposed changes:
● Scalability
○ Scales to millions of tables/columns via page-based storage.
○ Efficient caching through buffer manager integration.
○ Supports 4 billion unique objects using the OID system.
● Performance
○ Reduced disk I/O with in-memory catalog cache.
○ Fast constraint checks (O(log n)) using index-backed lookups.
○ Minimized fragmentation with slotted page layout.
● Consistency
○ Maintains referential integrity via foreign key constraints.
○ Prevents partial schema changes with transaction support.
○ Ensures durability through a write-through cache.
● Extensibility
○ Modular constraint system allows for future constraint types.
○ Supports user-defined types using the pg_type table.
○ Generic page format supports adding system catalogs.
● Standards Compliance
○ Based on PostgreSQL's proven catalog architecture.
○ Adheres to standard SQL constraint semantics.
○ Uses industry-standard 8KB page size.

3. Backend Functions
3.1 Functions to be Created
3.1.1 Catalog Initialization Functions
bootstrap_catalog() -> <(), CatalogError>
Description: Initialize the self-hosting catalog system with system tables
Implementation:
● Setup Storage: Create database/global/catalog_pages/, initialize OID counter at
10000, and create system catalog files (pg_database, pg_table, pg_column,
pg_constraint, pg_index, pg_type).
● Initialize Files: For each catalog file, create it, initialize page 0 as header, and
allocate the first data page.
● Bootstrap Metadata: Insert self-describing entries into pg_table and pg_column
for all system catalogs.
● Insert Core Data: Add built-in types (INT, BIGINT, FLOAT, etc.) into pg_type and
create the system database entry (db_oid=1, name="system").
● Finalize & Verify: Enable bootstrap_mode and verify setup by querying pg_table
to ensure all catalogs are registered.
init_catalog_page_storage(buffer_manager: &mut BufferManager) ->
Result<CatalogPageManager, CatalogError>
Description: Initialize page-based catalog storage backend
Implementation:
● Instantiate Manager: Create CatalogPageManager instance.
● Register Paths: Map catalog names (pg_database, pg_table, etc.) to their file
paths in database/global/catalog_pages/.
● Check Files: For each catalog file, verify whether it exists.
● Initialize or Bootstrap: If file exists, load page count from header; if not, call
bootstrap_catalog().
● Return Manager: Provide the fully initialized page manager.
3.1.2 Catalog Initialization Functions
register_builtin_types(catalog: &mut Catalog) -> Result<(), CatalogError>
Description: Register all built-in data types in pg_type
Implementation:
● Iterate Types: Process each built-in type.
● Serialize: Convert type metadata to bytes using serialize_type_tuple(type).
● Insert: Store in pg_type via page_manager.insert_catalog_tuple("pg_type",
bytes).
● Cache: Add type to in-memory cache (cache.insert_type(type_oid, type)).
● Complete: Return Ok(()) after successful insertion.
lookup_type_by_name(catalog: &Catalog, type_name: &str) -> Result<DataType,
CatalogError>
Description: Resolve type name to DataType metadata
Implementation:
● Check Cache First: Call cache.get_type_by_name(type_name) and return if
found.
● Scan Catalog: If not cached, scan pg_type using
page_manager.scan_catalog("pg_type").
● Deserialize Tuples: Convert each tuple using deserialize_type_tuple(bytes).
● Match & Cache: If type_name matches, cache the entry and return it.
● Handle Miss: If no match found, return CatalogError::TypeNotFound.
calculate_tuple_size(columns: &[Column], types: &HashMap<u32, DataType>) ->
(usize, bool)
Description: Calculate fixed and variable portions of tuple size
Implementation:
● Initialize: Set fixed_size = 0 and has_variable = false.
● Iterate Columns: For each column, fetch its type using
types.get(&column.type_oid).
● Handle Fixed Types: If type_length > 0, add it to fixed_size.
● Handle Variable Types: If type_length == -1, set has_variable = true and add 2
bytes (length prefix) to fixed_size.
● Return Result: Output (fixed_size, has_variable).
3.1.3 Constraint Management Functions
add_primary_key_constraint(catalog: &mut Catalog, table_oid: u32, column_oids:
Vec<u32>, constraint_name: Option<String>) -> Result<u32, CatalogError>
Description: Add a primary key constraint to a table
Implementation:
● Validate Preconditions: Ensure table exists and no existing primary key (scan
pg_constraint for table_oid + PrimaryKey).
● Validate & Update Columns: Confirm each column exists and set is_nullable =
false in pg_column.
● Prepare Metadata: Generate constraint name if missing, allocate a new
constraint OID.
● Create Backing Index: Create a unique B-tree index (is_unique=true,
is_primary=true) and obtain index_oid.
● Persist & Refresh: Build Constraint struct, serialize and insert into pg_constraint,
invalidate table cache, and return constraint_oid.
add_foreign_key_constraint(catalog: &mut Catalog, table_oid: u32, column_oids:
Vec<u32>, referenced_table_oid: u32, referenced_column_oids: Vec<u32>,
on_delete: ReferentialAction, on_update: ReferentialAction, constraint_name:
Option<String>) -> Result<u32, CatalogError>
Description: Add a foreign key constraint
Implementation:
● Validate Tables & Columns: Ensure referencing and referenced tables exist, and
column counts match.
● Check Referenced Integrity: Confirm referenced columns are covered by a
PRIMARY KEY or UNIQUE constraint in pg_constraint.
● Verify Type Compatibility: For each column pair, fetch type_oid from pg_column
and ensure types match.
● Prepare Constraint: Generate name if missing, allocate constraint OID, and build
Constraint struct with ForeignKey metadata.
● Persist & Refresh: Serialize and insert into pg_constraint, invalidate cache, and
return constraint_oid.
add_unique_constraint(catalog: &mut Catalog, table_oid: u32, column_oids:
Vec<u32>, constraint_name: Option<String>) -> Result<u32, CatalogError>
Description: Add a unique constraint to a table
Implementation:
● Validate Inputs: Ensure table and specified columns exist.
● Create Index: Create a unique B-tree index (is_unique=true, is_primary=false).
● Allocate OID: Generate a new constraint OID.
● Build & Persist: Create Constraint struct with Unique metadata, serialize, and
insert into pg_constraint.
● Refresh & Return: Invalidate cache and return constraint_oid.
add_not_null_constraint(catalog: &mut Catalog, table_oid: u32, column_oid: u32)
-> Result<(), CatalogError>
Description: Add a NOT NULL constraint to a column
Implementation:
● Validate Inputs: Ensure the table and column exist.
● Update Metadata: Set is_nullable = false in pg_column for the given column_oid.
● Optional Constraint Record: Optionally create and insert a corresponding record
in pg_constraint.
● Invalidate Cache: Clear affected table metadata from cache.
● Return Result: Return Ok(()) upon success.
validate_constraints(catalog: &Catalog, table_oid: u32, tuple_data: &[u8]) ->
Result<(), ConstraintViolation>
Description: Validate all constraints for a tuple before insertion
Implementation:
● Prepare Context: Deserialize tuple and fetch table metadata (get_table_by_oid).
● Column Validation: Enforce NOT NULL constraints; return error if any
non-nullable column is NULL.
● Load Constraints: Retrieve all table constraints (get_constraints_for_table).
● Enforce Constraints:
○ UNIQUE / PRIMARY KEY: Check index for duplicates; return
UniqueViolation if found.
○ FOREIGN KEY: Verify referenced values exist; return ForeignKeyViolation
if missing.
○ CHECK: Evaluate expression (future support).
● Success: Return Ok(()) if all validations pass.
3.1.4 Index Management Functions
create_index(catalog: &mut Catalog, table_oid: u32, column_oids: Vec<u32>,
is_unique: bool, is_primary: bool, index_name: Option<String>) -> Result<u32,
CatalogError>
Description: Create a B-tree index on specified columns
Implementation:
● Validate & Prepare: Ensure table and columns exist; generate index name if
missing and allocate a new index_oid.
● Create Storage: Create (database/base/{db_name}/indexes/{index_name}.idx)
index file and initialize B-tree root page.
● Build Metadata: Construct Index struct (BTree, column_oids, uniqueness flags,
initial root page).
● Persist Catalog Entry: Serialize and insert index record into pg_index.
● Populate Index: Scan table tuples, extract indexed values, insert into B-tree, and
return index_oid.
drop_index(catalog: &mut Catalog, index_oid: u32) -> Result<(), CatalogError>
Description: Drop an index from the system
Implementation:
● Load Metadata: Query pg_index to retrieve index details.
● Check Dependencies: Scan pg_constraint for matching index_oid; if referenced,
return error.
● Remove Storage: Delete the index file from disk.
● Update Catalog: Remove the corresponding row from pg_index.
● Refresh & Return: Invalidate cache and return Ok(()).
3.1.5 Database Operations Functions
create_database_enhanced(catalog: &mut Catalog, db_name: &str, owner: &str,
encoding: Encoding) -> Result<u32, CatalogError>
Description: Enhanced database creation with metadata
Implementation:
● Validate & Allocate: Ensure database name is non-empty and unique; allocate a
new db_oid.
● Create Storage: Create directory database/base/{db_name}.
● Build Metadata: Construct Database struct with owner, encoding, timestamp,
and empty tables map.
● Persist Catalog Entry: Serialize tuple ([oid][owner][name][created_at][encoding])
and insert into pg_database.
● Update Memory & Return: Add to in-memory map and cache, then return
db_oid.
drop_database(catalog: &mut Catalog, db_name: &str) -> Result<(), CatalogError>
Description: Enhanced database creation with metadata
Implementation:
● Locate Database: Fetch database from catalog.databases and obtain db_oid.
● Drop All Tables: Query pg_table for tables with matching db_oid and call
drop_table for each.
● Remove Catalog Entry: Delete the corresponding row from pg_database.
● Delete Storage: Remove database directory (database/base/{db_name}).
● Cleanup & Return: Remove from in-memory map, invalidate cache, and return
Ok(()).
3.1.6 Table Operations Functions
create_table_enhanced(catalog: &mut Catalog, db_name: &str, table_name: &str,
columns: Vec<ColumnDefinition>, constraints: Vec<ConstraintDefinition>) ->
Result<u32, CatalogError>
Description: Enhanced table creation with constraints
Implementation:
● Validate & Allocate: Ensure database exists and table name is unique; allocate
table_oid.
● Process Columns: For each column: allocate column_oid, resolve type, build
Column struct, serialize, and insert into pg_column.
● Create Storage & Metadata: Create table file, initialize header + first data page,
build Table struct (stats, constraints, indexes).
● Persist & Apply Constraints: Serialize and insert into pg_table; process constraint
definitions (PK, FK, UNIQUE, NOT NULL) via respective functions.
● Update Memory & Return: Add table to database map, cache metadata, and
return table_oid.
drop_table(catalog: &mut Catalog, table_oid: u32) -> Result<(), CatalogError>
Description: Drop table and all dependent objects
Implementation:
● Validate & Check Dependencies: Load table metadata and ensure no foreign
keys reference it (pg_constraint check).
● Drop Indexes: Query pg_index for the table and call drop_index for each.
● Remove Constraints & Columns: Delete related rows from pg_constraint and
pg_column.
● Delete Storage & Catalog Entry: Remove table file and delete corresponding row
from pg_table.
● Cleanup & Return: Remove from in-memory database map, invalidate cache,
and return Ok(()).
alter_table_add_column(catalog: &mut Catalog, table_oid: u32, column_def:
ColumnDefinition) -> Result<u32, CatalogError>
Description: Add a new column to an existing table
Implementation:
● Validate & Allocate: Ensure table exists and column name is unique; allocate
column_oid and determine new column position.
● Prepare Column: Resolve type, build Column struct, serialize, and insert into
pg_column.
● Update Table Metadata: Append column to table.columns and update
last_modified timestamp.
● Handle Defaults & Constraints: If NOT NULL without default, return error; if
default exists, update all existing rows.
● Finalize: Invalidate cache and return column_oid.
3.1.7 Catalog Query Functions
get_table_metadata(catalog: &Catalog, db_name: &str, table_name: &str) ->
Result<TableMetadata, CatalogError>
Description: Retrieve complete table metadata including columns and constraints
Implementation:
● Check Cache First: Attempt to retrieve table metadata from cache; return if
found.
● Resolve Identifiers: If not cached, get db_oid from pg_database and fetch
table_oid from pg_table.
● Load Components: Query pg_column (ordered), pg_constraint, and pg_index for
the table.
● Construct Metadata: Build TableMetadata struct from retrieved data.
● Cache & Return: Store in cache and return the metadata.
get_constraints_for_table(catalog: &Catalog, table_oid: u32) ->
Result<Vec<Constraint>, CatalogError>
Description: Get all constraints for a table
Implementation:
● Check Cache First: Attempt to retrieve constraints from
catalog.cache.get_constraints(table_oid); return if found.
● Query Catalog: If not cached, scan pg_constraint for matching table_oid.
● Deserialize: Convert each constraint tuple into a Constraint struct.
● Cache Results: Store retrieved constraints in cache.
● Return Data: Return the list of constraints.
3.2 Functions to be Modified
3.2.1 init_catalog() -> Result<(), CatalogError>
Current Implementation: The system creates database/global/catalog.json if it does
not exist and initializes it with an empty JSON structure: {"databases": {}}.
Updated Functionality: The system now supports dual-mode initialization by
maintaining legacy JSON compatibility while detecting whether page-based system
catalogs exist and calling bootstrap_catalog() if they do not.
Reason for Update: This enables migration from JSON-based storage to page-based
storage while maintaining backward compatibility during the transition period.
Implementation:
● Check Page Storage: If database/global/catalog_pages/ exists, initialize
page-based backend and load system catalogs into memory.
● Fallback to Legacy: If not, check for catalog.json; load JSON catalog (legacy
mode) and optionally migrate.
● Bootstrap if Needed: If neither exists, call bootstrap_catalog() to create system
catalogs and register built-in types.
● Initialize Core State: Load OID counter (pg_oid_counter.dat) and set up catalog
cache.
● Finalize: Complete initialization and return Ok(()).
3.2.2 load_catalog(buffer_manager: &mut BufferManager) -> Result<(),
CatalogError>
Current Implementation: The system reads catalog.json from disk, deserializes the
JSON into a Catalog struct, and returns an empty catalog if an error occurs.
Updated Functionality: The system now loads catalog metadata from system catalog
pages and populates an in-memory cache with frequently accessed entries.
Reason for Update: The update supports the transition to page-based storage while
improving performance through caching.
Implementation:
● Initialize Core Components: Create empty Catalog, load OID counter, and
initialize page manager with buffer_manager.
● Load Databases: Scan pg_database, deserialize each tuple, and populate
catalog.databases.
● Load Tables per Database: For each database, scan pg_table by db_oid and
deserialize tables.
● Load Table Dependencies: For each table, query and attach pg_column,
pg_constraint, and pg_index data.
● Warm Cache & Return: Populate cache with initial entries (e.g., first 100) and
return the fully built catalog.
3.2.3 save_catalog(catalog: &Catalog) -> Result<Catalog, CatalogError>
Current Implementation: The system serializes the Catalog to JSON and writes it to
catalog.json.
Updated Functionality: The system now persists catalog changes by flushing dirty
catalog pages to disk and performing incremental updates that write only modified
entries instead of the entire catalog.
Reason for Update: Page-based storage removes the need for full catalog serialization
and improves performance for large catalogs.
Implementation:
● Flush Buffers: Call buffer_manager.flush_pages() to write all dirty pages to disk.
● Persist OID Counter: Save current OID state to pg_oid_counter.dat.
● Sync Catalog Files: Ensure all catalog files are fully synced to disk.
● Optional Backup: Write JSON backup for backward compatibility (if enabled).
● Complete: Return Ok(()).
3.2.4 create_database(catalog: &mut Catalog, db_name: &str) -> Result<u32,
CatalogError>
Current Implementation: The system validates the database name, inserts it into the
in-memory catalog, writes the catalog to JSON, and creates the corresponding
directory.
Updated Functionality: The system now calls create_database_enhanced() with a
default owner and UTF-8 encoding and persists the database entry by inserting it into
pg_database in the page-based catalog.
Reason for Update: The update enables support for additional metadata such as
owner, encoding, and timestamp while integrating database creation with the
page-based catalog system.
Implementation:
● Invoke Creation: Call create_database_enhanced(catalog, db_name,
"default_user", Encoding::UTF8).
● Handle Initialization: Let the enhanced function perform validation, allocation,
and setup.
● Create Metadata & Storage: Database directory and catalog entries are created
internally.
● Update Catalog State: In-memory structures and cache are updated.
● Return Result: Return the generated db_oid.
3.2.5 create_table(catalog: &mut Catalog, db_name: &str, table_name: &str,
columns: Vec<ColumnDefinition>, constraints: Vec<ConstraintDefinition>)
-> Result<u32, CatalogError>
Current Implementation: The system validates the database and table name, creates
a Table struct with columns, inserts it into the in-memory catalog, saves the catalog to
JSON, and creates the table file initialized with init_table().
Updated Functionality: The system now accepts and validates constraint definitions
for consistency and persists metadata in a page-based catalog by inserting entries
into pg_table and pg_column.
Reason for Update: The update enables support for primary keys, foreign keys,
unique, and NOT NULL constraints while integrating table creation with the
page-based catalog system.
Implementation:
● Invoke Creation: Call create_table_enhanced(catalog, db_name, table_name,
columns, constraints).
● Handle Validation: Let the enhanced function validate database, table name,
columns, and constraints.
● Create Storage & Metadata: Table file and catalog entries are created internally.
● Apply Constraints & Indexes: Primary key, foreign key, unique, and other
constraints are processed.
● Return Result: Return the generated table_oid.
3.2.6 show_databases(catalog: &Catalog) -> Result<(), CatalogError>
Current Implementation: The system iterates over catalog.databases.keys() and prints
the database names.
Updated Functionality: The system now queries pg_database in the page-based
system catalog and displays additional metadata such as owner, created_at, and
encoding.
Reason for Update: The update makes the page-based catalog the source of truth
and provides richer database information.
Implementation:
● Scan Catalog: Call catalog.page_manager.scan_catalog("pg_database").
● Deserialize Tuples: Convert raw tuple bytes into Database structs.
● Format Output: Extract db_name | owner | created_at | encoding.
● Print Results: Display rows (e.g., mydb | default_user | ... | UTF-8).
● Return: Finish with Ok(()).
3.2.7 show_databases(catalog: &Catalog, db_name: &str) -> Result<(),
CatalogError>
Current Implementation: The system retrieves the database from catalog.databases,
iterates over database.tables.keys(), and prints the table names.
Updated Functionality: The system now queries pg_table, filters results by db_oid, and
displays additional metadata such as row count, page count, and created_at.
Reason for Update: The update uses the page-based catalog as the authoritative
source and provides table-level statistics.
Implementation:
● Resolve Database ID: Fetch db_oid from pg_database.
● Scan Tables: Query pg_table using db_oid.
● Deserialize Tuples: Convert raw bytes into Table structs.
● Format & Print: Display table_name | row_count | page_count | created_at.
● Return: Complete operation with Ok(()).
