0. Executive Summary
The Catalog Manager has been transformed from a basic JSON-based metadata system into a production-grade, page-based self-hosting catalog system with full constraint management, index metadata storage, and buffer manager integration. The implementation closely follows PostgreSQL's proven catalog architecture while maintaining consistency with RookDB's existing infrastructure.

1. Architecture Overview
Page-Based Catalog Structure:
database/global/
├── catalog_pages/          # NEW: System catalog files
│   ├── pg_database.dat     # Database metadata (OID, name, owner, encoding, created_at)
│   ├── pg_table.dat        # Table metadata (OID, name, parent DB, row/page counts)
│   ├── pg_column.dat       # Column metadata (OID, position, type, nullable, defaults)
│   ├── pg_constraint.dat   # Constraint metadata (OID, type, referenced tables/actions)
│   ├── pg_index.dat        # Index metadata (OID, type, indexed columns, uniqueness)
│   ├── pg_type.dat         # Data type definitions (OID, category, alignment, length)
│   ├── pg_oid_counter.dat  # Persistent OID allocation counter
│   └── catalog.json        # DEPRECATED: Dual-mode legacy support
└── base/
  └── {database}/
      └── {table}.dat      # User table data files
	

This enables buffer manager caching, consistent page format across system and user tables, and self-hosting capability.

2. Core Features Implemented


2.1 Catalog Page Manager
* Bridge between logical catalog operations and page-based storage
* Manages catalog file creation and allocation
* CRUD operations for catalog tuples


Buffer Manager Integration: All catalog page operations use the buffer manager with full pin/unpin semantics:
* insert_catalog_tuple(): Uses bm.pin_page() + bm.unpin_page() with is_dirty=true
* read_catalog_tuple(): Uses bm.pin_page() to load, unpin when done
* scan_catalog(): Pins each page, unpins after deserializing all tuples
* update_catalog_tuple(): Pins, deletes old, inserts new, unpins with dirty flag


The Buffer Manager itself was enhanced to support this integration:
pub struct BufferManager {
   pages: Vec<Box<Page>>,
   page_metadata: HashMap<PageId, PageMetadata>,
   lru_queue: VecDeque<PageId>,
   max_pages: usize,
}

pub struct PageMetadata {
   pin_count: u32,           // Prevents eviction while in use
   is_dirty: bool,           // Write-back on unpin
   last_accessed: u64,       // LRU tracking
   buffer_index: usize,      // Location in buffer pool
}
	

Catalog Bootstrapping: The page manager handles system catalog initialization; from system table creations (pg_database, pg_table, pg_column, pg_constraint, pg_index, pg_type) to metadata initialization (each system table has its own row in pg_table, columns in pg_column). Appropriate OIDs are also allocated for all system objects. This enables the catalog to describe itself, eliminating any external schema dependency.


Some key design choices made:
Area
	Problem Description
	Implemented Solution
	Catalog File Header
	The initial 8-byte header prevented correct disk seeks, which expect 8 KB page sizes.
	Delegated initialization to init_table(), ensuring a complete 8 KB header page is created alongside the 8 KB data page.
	update_catalog_tuple
	Variable-length strings could not be updated in place, leading to a length mismatch.
	The tuple is deleted and then re-inserted, with the function returning the new (page_num, slot_id) location.
	insert_catalog_tuple
	The slot ID was incorrectly hardcoded to 0 instead of reflecting the actual insertion slot on the page.
	The correct slot_id is now computed post-insertion using the page's lower pointer: slot_id = (lower - HEADER_SIZE) / ITEM_ID_SIZE - 1.
	OID Counter Persistence
	The OID counter was not written to disk, causing it to reset on every restart and leading to OID collisions.
	Catalog::alloc_oid() now writes the counter state to pg_oid_counter.dat whenever the page backend is active.
	

2.2 Comprehensive Constraint System
Four Constraint Types Implemented:
1. Primary Key (PrimaryKey): Enforces uniqueness on one or more columns. Automatically NOT NULL. Creates backing unique index via ‘pg_index’. Supports composite keys (e.g., ‘PRIMARY KEY (col1, col2)’).
2. Foreign Key (ForeignKey): Referential integrity between tables. Supports composite foreign keys. Prevents orphaned rows via validation in ‘load_csv.rs’. Allows following referential actions:
* NO ACTION (default): Reject DELETE/UPDATE if child rows exist
* CASCADE: Propagate DELETE/UPDATE to all child rows
* SET NULL: Set child columns to NULL on parent DELETE/UPDATE
* RESTRICT: Like NO ACTION but checked before transaction start
3. Unique Constraint (Unique): Enforces uniqueness on one or more columns. Allows NULL values (unlike PRIMARY KEY). Supports multiple UNIQUE constraints per table. Composite unique keys fully supported.
4. Not-Null Constraint (NotNull): Column-level enforced during INSERT/UPDATE. Blocks NULL insertions at validation stage. Checked before writing to storage
  

The core functions implementing these are:
* add_primary_key_constraint(): Creates PK with unique index
* add_foreign_key_constraint(): Validates reference tables, stores metadata
* add_unique_constraint(): Creates unique constraint with backing index
* add_not_null_constraint(): Column-level NULL blocking
* validate_constraints(): Real-time validation during INSERT
* check_referential_integrity(): Cross-table reference integrity


2.3 Index Metadata Management
Key features include:
* Index Types Supported: B-Tree (primary), Hash (framework ready)
* Metadata Tracking: Index OID, name and type. Indexed column list (supports multi-column indexes). Unique flag and primary key flag. Page count for statistics.
* Integration with constraints: PK and UNIQUE automatically create indexes, which are stored persistently in ‘pg_index.dat’.


The core functions implementing these are:
* create_index(): Register index in `pg_index`, create backing file
* drop_index(): Remove index metadata and physical file
* get_indexes_for_table(): Retrieve all indexes on a table
* get_indexes_by_columns(): Find indexes that cover given columns
* find_or_create_index(): Reuse existing or create new unique index


2.4 In-Memory LRU Catalog Cache
Key features include:
* LRU Eviction: Keeps hot catalog entries in memory
* Entry Types Cached: Databases (by name), Tables (by database OID + name), Constraints (by table OID), Indexes (by table OID) and Data Types (by type OID)
* Invalidation Strategy: Ensures cache consistency with page-based storage. Invalidated on all DDL operations..
* Performance Impact: Reduces disk I/O for frequently accessed metadata
* Default cache size is 256 entries (tunable via ‘new(capacity)’); and access-order is tracked for LRU replacement
Cache is instantiated in the `Catalog` struct and invalidation calls are wired throughout all DDL operations.

2.6 OID (Object Identifier) System
The main features of the system are as follows:
* Persistent OID Counter stored in ‘pg_oid_counter.dat’
* OID Ranges: System OIDs (0-9999, reserved for built-in objects), User OIDs (10K+, dynamically allocated)
* Thread-Safe Allocation: ‘Catalog::alloc_oid()’ increments and persists
* OID Persistence: Counter written to disk on every allocation (when page backend active)
* Prevents Collisions: On restart, loads persisted counter value


Implemented Tuple Types:        
* serialize_database_tuple() / deserialize_database_tuple()
* serialize_table_tuple() / deserialize_table_tuple()
* serialize_column_tuple() / deserialize_column_tuple()
* serialize_constraint_tuple() / deserialize_constraint_tuple()
* serialize_index_tuple() / deserialize_index_tuple()
* serialize_type_tuple() / deserialize_type_tuple()


The final result is a system with following key attributes:
* Supports 4 billion unique objects (u32 range)
* Each database, table, column, constraint, index gets unique OID
* Enables foreign key references within system catalogs (self-hosting)
* Matches PostgreSQL's proven OID system


2.7 Serialization / Deserialization
Purpose: Binary representation of catalog tuples for page storage


Key features are:
* Handles all data types (INT, VARCHAR, arrays of u32 for OIDs, etc.)
* Maintains field order matching system catalog schema
* Length prefixes for variable-length fields


The updated tuple format uses a fixed-length header + variable-length payload:
* [tuple_header: u32] [fixed-length fields] [var-length offsets] [var-length data]

3.1 Column Structure Extension
From (Legacy Code)
struct Column {
  name: String,
  data_type: String,  // Simple string like "INT"
}
	

To (Updated Code)
pub struct Column {
  pub column_oid: u32,
  pub name: String,
  pub column_position: u16, // 1-based position
  pub data_type: DataType, // Full type metadata
  pub type_modifier: Option<TypeModifier>,  // Qualifiers like VARCHAR(100)
  pub is_nullable: bool,
  pub default_value: Option<DefaultValue>, // NULL, integer, string, CURRENT_TIMESTAMP
  pub constraints: Vec<u32>,          // Constraint OIDs
}
	

Key benefits include:
* Precise type information (length, alignment, category)
* Default value support for INSERT defaults
* Constraint linkage via OIDs
* Ordered column positions for deterministic layout

5. Known Limitations & Future Work


5.1 Partially Implemented
* B-Tree Indexing: Currently uses a linear-scan index validation (functionally correct), but will transition to a true B-Tree for O(log n) lookups. No correctness issues, just slower performance on large tables for now.


5.2 Deferred Features
* Query optimizer using index statistics
* Constraint deferral within transactions
* CHECK constraints (expression parsing deferred)
* Constraint name resolution (using OIDs instead)
* Tuple-level row versioning (MVCC)
